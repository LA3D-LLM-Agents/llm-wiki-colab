# Build notes — llm-wiki plugin (v1 skeleton)

Scope: **v1 = Claude Code adapter only** (MCP server, Cursor adapter, and
features-as-plugins are deferred). Ported from
`crcresearch/llm-wiki-memory-template`. Design: llm-wiki-vision wiki,
*Plugin-Packaging-Architecture*.

## Done (this scaffold)

- Monorepo structure: `core/`, `adapters/claude-code/{commands,skills,hooks,.claude-plugin}`, `.claude-plugin/marketplace.json`.
- Ported verbatim from upstream: gates (`core/agents/*`), `core/scripts/{lib,wiki-write-protocol,kg}`, `core/init-wiki.sh`, `core/Edge-Types.md.template`, `core/templates/*`, the three commands and three skills.
- Authored: `plugin.json`, `marketplace.json`, `hooks.json`, `/wiki-init` and `/wiki-doctor` command specs, `README.md`, `package.sh`.
- Hooks repathed to the `.llm-wiki/` + opt-in model:
  - `hooks/session-start.sh`: **rewritten** — silent no-op unless `.llm-wiki/` exists (opt-in), derives the namespace from `origin` at runtime (no install-time `${REPO_NAME}` bake), emits orientation + guidance + index + last-5 log against `.llm-wiki/`.
  - `hooks/ensure-wiki.py`: `wiki_rel` repathed to `.llm-wiki` (keeps the production clone/attach: VCS-aware, atomic staged clone, `--ff-only` safety).
  - `hooks/posttooluse.sh`: matcher scoped to `.llm-wiki/*.md`; gate reference points at the plugin core.
- `core/init-wiki.sh` adapted for the plugin model and **create-mode smoke-tested**: lib-source path fixed, `WIKI_DIR` → `.llm-wiki/`, `CLAUDE.md` writer removed, WIKI-INDEX registration removed, `.gitignore` line auto-added. Verified in a fresh repo: builds `.llm-wiki/` with namespaced files; the only project footprint is the `.gitignore` line; no `CLAUDE.md` or `WIKI-INDEX` written.
- **State-2 (`--github`) attach validated** end to end against the real `chrissweet/llm-wiki-vision` GitHub wiki. `ensure-wiki.py` (SessionStart auto-clone) cleanly clones the fully-populated wiki into `.llm-wiki/`: silent on success, no scaffolding, zero tracked-file changes. `/wiki-init --github` also attaches and adds the gitignore line (with the create-mode caveat in TODO 1).
- **Install validated**: `claude plugin validate` passes for both the marketplace and the plugin manifest; a local `marketplace add` → `install` → `details` → `uninstall` cycle confirms the plugin loads enabled with 5 commands, 3 skills, and both hooks (SessionStart + PostToolUse) registered (~438 always-on tokens).
- **Packaging**: `core/` is now committed into `adapters/claude-code/core/` (not gitignored) so the marketplace install is self-contained; root `core/` stays the edit-canonical copy and `package.sh` re-syncs it. Only the KG `build/` output is ignored.
- **State-2 SCHEMA-safe attach**: `init-wiki.sh --github` re-detects mode after the clone, so attaching to an already-populated wiki runs *update* mode (append-missing-only), never overwriting a customized SCHEMA. Verified against llm-wiki-vision (switches to update mode; no tracked-file changes).
- **`ensure-wiki.py` is a strict opt-in SessionStart hook**: if `.llm-wiki/` is present it fast-forwards it (safe `--ff-only`) and ensures the gitignore line; if absent it stays silent and does **not** auto-clone, even when the repo has a GitHub wiki. Attaching a repo to a wiki is `/wiki-init`'s job only, matching the documented opt-in model (folder presence = opt-in). The earlier auto-clone-from-remote behavior (inherited from the template, where every instance was meant to carry a wiki) was removed because it broke the opt-in guarantee: a globally-installed plugin would otherwise create a `.llm-wiki/` in any repo that happens to have a GitHub wiki. State-3 ("no wiki yet, create the first page") now lives only in `/wiki-init`.
- **Test suite added** (`tests/`, repo-only, not part of the shipped plugin): `tests/run.sh` assembles the adapter then runs L1 manifest validation (`claude plugin validate` + JSON) and path-integrity, L2 hook unit tests (opt-in silence, last-5-log, PostToolUse scoping, `ensure-wiki` no-auto-clone), L3 `/wiki-init` create-mode footprint, and the 10 `wiki-write-protocol` scenarios (ported to `tests/wiki-write-protocol/`, `protocol.sh` symlinked from `core/`). All green locally and in CI. `.github/workflows/ci.yml` runs the suite plus a packaging drift guard (re-runs `package.sh`, fails on a dirty adapter `core/`). The runner retries the timing-sensitive protocol suite once so an occasional race-scenario flake does not red CI (the deterministic `test_*.sh` are never retried).
- **Guidance rewritten**: `core/templates/guidance.md` (the SessionStart-injected memory-boundary / wiki-maintenance text) was rewritten for the plugin context: no `CLAUDE.md`/setup.sh sentinels, `.llm-wiki/` paths, plugin gate and command references, no `${REPO_NAME}` placeholders.
- **`/wiki-doctor` is a real script, not just a spec**: `core/scripts/wiki-doctor.sh` runs the 7-point checklist (plugin root, gates, hooks, `.llm-wiki/` attachment, KG deps, remote reachability, orientation dry-run). Structural checks are pass/fail; missing KG deps or no network are warnings, so a clean install still exits 0. The command wraps it, `test_doctor.sh` asserts a healthy install reports zero structural failures, and path-integrity checks the script ships. (The original spec's "`wiki_push` dry-run" check was replaced with a reachability + clean-tree check, since `wiki_push` has no dry-run.)

- **Porting-discipline follow-up (2026-07-22)**: after a sanity check on port-vs-reinvent, ported `core/scripts/wiki-reciprocity.py` verbatim (wired into `/wiki-lint`, with `test_reciprocity.sh` + fixtures) and the upstream self-contained `update_mechanics_test.py` (exercises `ensure-wiki.py`'s `update_wiki` fast-forward mechanics, `test_mechanics.sh`, 19 checks, real added coverage). Also fixed the ported commands/skills, which had carried **unsubstituted `{{REPO_NAME}}`**, `wiki/<repo>.wiki/`, `wiki/agents/`, and `scripts/` references (a too-literal port); adapted to `.llm-wiki/` and `${CLAUDE_PLUGIN_ROOT}/core/...`. The upstream test *harness* (`run.sh`/`sandbox.sh`/`template.sh` + patch/assertions) stays un-ported by design: it is welded to the template's instantiate/adopt model, so the plugin's runner is legitimately fresh.
- **Attach is now idempotent, divergence quirk fixed (2026-07-22)**: `init-wiki.sh` update-mode (attaching to an existing wiki) previously rewrote the SCHEMA, stamped pages, and appended a gratuitous "frontmatter convention" log entry, then committed, all un-pushed local commits that diverged a clone from the pushed wiki (seen when a session's `.llm-wiki/` drifted from the remote). Attach is now clone + gitignore + report, with **zero commits**; migrations/convention drift are a `/wiki-lint` concern, not a silent attach side-effect. `test_attach_idempotent.sh` codifies it (a second `/wiki-init` leaves `HEAD` unchanged and the tree clean).
- **Federation `ask` ported (2026-07-22)**: `core/scripts/agent-comms/ask.sh` (verbatim, self-contained, clones remote wikis to a cache and self-injects the Query preamble, so no `.llm-wiki/` adaptation needed) and `enroll.sh` (repathed to `.llm-wiki/`). Added the `/wiki-ask` command (federation discovery + clone-and-invoke; the agent is the picker, direct mode only) and `/wiki-enroll` (publish an agent Card + the `nd-llm-wiki` discovery topic). `test_ask.sh` smoke-tests usage/errors plus one fully offline direct-mode resolve -> clone -> invoke; `/wiki-doctor` gained a `jq`/`curl`/`gh` deps check. This is the *federated* ask (the whole "how do I answer a question from another agent's wiki" tier), already built upstream.
